71 /* The __CONCAT macro is used to concatenate parts of symbol names, e.g. with
72  * "#define OLD(foo) __CONCAT(old,foo)", OLD(foo) produces oldfoo. The __CONCAT
73  * macro is a bit tricky to use if it must work in non-ANSI mode -- there must
74  * be no spaces between its arguments, and for nested __CONCAT's, all the
75  * __CONCAT's must be at the left. __CONCAT can also concatenate double-quoted
76  * strings produced by the __STRING macro, but this only works with ANSI C.
77  *
78  * __XSTRING is like __STRING, but it expands any macros in its argument first.
79  * It is only available with ANSI C. */
80

237 /* GNU C version 2.96 adds explicit branch prediction so that the CPU back-end
238  * can hint the processor and also so that code blocks can be reordered such
239  * that the predicted path sees a more linear flow, thus improving cache
240  * behavior, etc.
241  *
242  * The following two macros provide us with a way to utilize this compiler
243  * feature. Use __predict_true() if you expect the expression to evaluate to
244  * true, and __predict_false() if you expect the expression to evaluate to
245  * false.
246  *
247  * A few notes about usage:
248  * * Generally, __predict_false() error condition checks (unless
249  * you have some _strong_ reason to do otherwise, in which case
250  * document it), and/or __predict_true() `no-error' condition
251  * checks, assuming you want to optimize for the no-error case.
252  *
253  * * Other than that, if you don't know the likelihood of a test
254  * succeeding from empirical or other `hard' evidence, don't
255  * make predictions.
256  *
257  * * These are meant to be used in places that are run `a lot'.
258  * It is wasteful to make predictions in code that is run
259  * seldomly (e.g. at subsystem initialization time) as the
260  * basic block reordering that this affects can often generate
261  * larger code. */
263 #define __predict_true(exp) __builtin_expect((exp), 1)
264 #define __predict_false(exp) __builtin_expect((exp), 0)
