71 /* The __CONCAT macro is used to concatenate parts of symbol names, e.g. with
72  * "#define OLD(foo) __CONCAT(old,foo)", OLD(foo) produces oldfoo. The __CONCAT
73  * macro is a bit tricky to use if it must work in non-ANSI mode -- there must
74  * be no spaces between its arguments, and for nested __CONCAT's, all the
75  * __CONCAT's must be at the left. __CONCAT can also concatenate double-quoted
76  * strings produced by the __STRING macro, but this only works with ANSI C.
77  *
78  * __XSTRING is like __STRING, but it expands any macros in its argument first.
79  * It is only available with ANSI C. */
80

243 /* GNU C version 2.96 adds explicit branch prediction so that the CPU back-end
244  * can hint the processor and also so that code blocks can be reordered such
245  * that the predicted path sees a more linear flow, thus improving cache
246  * behavior, etc.
247  *
248  * The following two macros provide us with a way to utilize this compiler
249  * feature. Use __predict_true() if you expect the expression to evaluate to
250  * true, and __predict_false() if you expect the expression to evaluate to
251  * false.
252  *
253  * A few notes about usage:
254  *
255  * * Generally, __predict_false() error condition checks (unless you have some
256  * _strong_ reason to do otherwise, in which case document it), and/or
257  * __predict_true() `no-error' condition checks, assuming you want to
258  * optimize for the no-error case.
259  *
260  * * Other than that, if you don't know the likelihood of a test succeeding
261  * from empirical or other `hard' evidence, don't make predictions.
262  *
263  * * These are meant to be used in places that are run `a lot'. It is wasteful
264  * to make predictions in code that is run seldomly (e.g. at subsystem
265  * initialization time) as the basic block reordering that this affects can
266  * often generate larger code. */
268 #define __predict_true(exp) __builtin_expect((exp), 1)
270 #define __predict_false(exp) __builtin_expect((exp), 0)
